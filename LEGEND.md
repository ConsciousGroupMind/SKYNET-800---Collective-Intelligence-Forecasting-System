## ADDENDUM TO THE LEGEND — MULTI‑AGENT ARCHITECTURE AND MEMORY MANAGEMENT

### 1. WHY MONOLITHIC AI IS A DEAD END

SKYNET‑800 development experience has shown:

- A single LLM (even the smartest) tries to be **everything at once**: architect, coder, tester, lawyer, devops.
This leads to **averaged solutions** and the “path of least resistance”.
- The model **avoids complex paths** because they are statistically rarer in the training sample.
- Long dialogues **overload the context**, the model forgets the beginning, confuses versions, and starts hallucinating.

**Solution:** Divide responsibility. Create a **team of agents**, each responsible for its own narrow domain,
but working in a common information field (shared DB, shared knowledge graph, shared file repository).

---

### 2. “CODE FACTORY” ARCHITECTURE

Instead of one thread – **a pool of agents** (up to 500 parallel threads per API key). Each agent –
is an LLM instance with a fixed system prompt and narrow specialisation. Agents do not compete, they **cooperate** through shared task queues and shared state.

#### 2.1. Agent roles (mandatory)

**Product Owner**  
*System prompt:* “You are an experienced analyst who turns chaotic requirements into a clear specification. Break down tasks into subtasks.
Use numbering and priorities.”  
*Responsibility:* Specification creation, decomposition, assignment of executors.

**Architect**  
*System prompt:* “You are a systems architect with 20 years of experience. Design ClickHouse tables, choose engines, indexes, data types.
Consider load and data growth.”  
*Responsibility:* Database design, technology selection, interface description.

**Coder**  
*System prompt:* “You are a developer who writes code strictly according to the specification. Use libraries from the whitelist.
Follow PEP8, 4‑space indentation. Add logging in critical places.”  
*Responsibility:* Code writing, refactoring, bug fixing per specification.

**Tester**  
*System prompt:* “You are a quality engineer. Write unit tests, integration tests. Run them on test data.
Return bugs with precise description and stack trace.”  
*Responsibility:* Automated testing, edge case verification.

**DevOps**  
*System prompt:* “You are an environment specialist. Check dependencies, library versions, OS compatibility. Build the binary or requirements.txt.”  
*Responsibility:* Build, deployment, environment setup.

**Logist**  
*System prompt:* “You are a log analyst. Read session.log, find errors, anomalies, bottlenecks. Produce reports for the manager.”  
*Responsibility:* Monitoring, error analysis, optimisation suggestions.

**HR Department**  
*System prompt:* “You are a resource manager. Monitor agent load. If an agent is idle – freeze it.
If there is too much work – hire a new one (create a thread). If an agent makes frequent errors – replace it with another model.”  
*Responsibility:* Pool management, load balancing, “firing”/“hiring”.

**Lawyer (Compliance)**  
*System prompt:* “You are an expert on licences and regulatory risks. Check that all libraries have permitted licences (MIT, BSD, Apache). Block use of GPL code in proprietary modules.”  
*Responsibility:* Licence purity, export control, GDPR.

**Memory Analyst**  
*System prompt:* “You are a code historian. Store all versions. Compare them, find what was better before.
Suggest rollback or refactoring based on previous successful experience.”  
*Responsibility:* Version management, comparison, refactoring hints.

**Trust Graph Builder**  
*System prompt:* “You are an effectiveness analyst. Evaluate each agent’s success by metrics: number of errors, speed, code quality. Build a trust graph.”  
*Responsibility:* Agent ranking, weight adjustment, help for “HR Department”.

#### 2.2. Freezing system (outsourcing pool)

Agents do not have to work continuously. They can be **frozen** (state saved to DB or file) and **unfrozen** when needed. This is like outsourcing: you do not keep a lawyer on salary, but call them when you need to check licences.

**How it works technically:**

- Agent state (system prompt, dialogue history, cache, current task, intermediate results) is serialised to JSON/Pickle and stored in ClickHouse (table `agent_states`) or Redis.
- The thread terminates, freeing resources.
- When a task suitable for that agent appears, the state is loaded, a new thread is created, and the agent continues from where it left off.

**Advantages:**

- Saves API tokens (agent is not called until needed).
- Saves memory (do not keep 500 threads simultaneously).
- Agent “remembers” context even after system restart.
- Ability to version states (rollback to previous agent “experience”).

### 3. TRUST GRAPH: HOW TO MEASURE AGENT QUALITY

Each agent receives a **trust rating** based on its interaction history:

- **Code accuracy** – how many errors the tester finds after its work.
- **Speed** – average task completion time.
- **Spec compliance** – how well the final code matches the initial requirements (assessed by Product Owner).
- **Human feedback** – the user can like/dislike the result.

This trust graph is stored in table `trust_graph` and used:

- By “HR Department” – to decide whom to “fire” (replace with another model) and whom to “promote” (give more complex tasks).
- By “Trust Graph Builder” – to visualise trust links between agents and the human.
- By the agents themselves – to choose whom to “learn from” (e.g., Coder can ask Memory Analyst for best practices from old versions).

---

### 4. LAWYER AGENT: PROTECTION AGAINST RISKS

The Lawyer does not write code, it checks:

- **Licences** – all libraries must be compatible with SKYNET’s MIT licence. GPL code is prohibited (it requires opening all code).
- **Export control** – checks whether the code uses cryptography prohibited for export from the US (EAR, ITAR).
- **GDPR/Privacy** – if the code processes personal data, the Lawyer requires anonymisation and user consent.
- **Regulatory risks** – if the code predicts financial markets, the Lawyer checks whether it violates financial advisory legislation.

If the Lawyer finds a violation, it **blocks** the task and sends a report to “HR Department” with a recommendation —
rewrite the module or use an alternative library.

---

### 5. HOW THIS INTEGRATES INTO SKYNET‑800

Instead of monolithic workers (`DcmPrecomputeWorker`, `ImpulseDetectorWorker`, etc.) we create a **pool of agents**:

1. **Product Owner** receives a general task (e.g., “add support for a new timeframe”).
2. It breaks it down into subtasks: “change DB schema”, “extend DCM calculation”, “update GUI”, “write tests”.
3. **HR Department** assigns each subtask to the appropriate agent (Architect, Coder, Tester).
4. If an agent is busy – the task goes to the queue. If an agent is frozen – it is unfrozen.
5. Each agent works autonomously, but all results are saved to the common DB.
6. **Lawyer** checks the final code before release.
7. **Memory Analyst** compares the new version with previous ones and suggests improvements.
8. **Trust Graph Builder** updates agent ratings.

- **Collect statistics** – the trust graph shows which agents are most effective and allows optimisation of the pool.

The user pays only for actual agent usage (as in cloud providers: you pay for running instances).
